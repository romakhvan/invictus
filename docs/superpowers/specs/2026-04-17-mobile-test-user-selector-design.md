# Дизайн селектора тестовых пользователей для mobile

## Цель

Централизовать выбор тестового пользователя для mobile-тестов в одном месте, чтобы тесты и фикстуры сами не решали:

- какой сценарий пользователя нужен
- под каким номером телефона выполнять вход
- какие критерии в базе определяют подходящего кандидата
- должен ли CLI override иметь приоритет

Новый дизайн вводит единый селектор, который возвращает готовый объект контекста для запрошенного mobile-сценария.

## Проблема

Сейчас логика разбросана по нескольким местам:

- `tests/mobile/onboarding/test_client_onboarding.py`
- `tests/mobile/helpers/session_helpers.py`
- `src/repositories/users_repository.py`
- `tests/conftest.py`
- `tests/mobile/conftest.py`

Из-за этого возникают повторяющиеся проблемы:

- правила выбора сценария сложно быстро найти
- критерии выбора в БД концептуально размазаны между тестами и helper-слоем
- для онбординга и для сценариев под существующим пользователем используются разные точки входа
- добавление нового сценария требует менять несколько слоёв
- фикстуры знают слишком много о деталях репозиториев

## Рекомендуемый подход

Создать отдельный модуль:

- `src/repositories/mobile_test_users_repository.py`

Этот модуль станет единой точкой входа для выбора тестового пользователя или номера телефона под известный mobile-сценарий.

Он будет предоставлять:

- `MobileTestUserScenario`
- `TestUserContext`
- `MobileTestUserSelector`

Существующие низкоуровневые helper-функции из `src/repositories/users_repository.py` можно временно сохранить как строительные блоки, но решение о выборе сценария должно переехать в новый селектор.

## Публичный API

```python
selector = MobileTestUserSelector(db)

context = selector.select(
    scenario=MobileTestUserScenario.ONBOARDING_NEW_USER,
    override_phone=onboarding_phone,
)
```

Селектор возвращает объект `TestUserContext`, а не просто строку с номером телефона.

## TestUserContext

`TestUserContext` — это dataclass, который содержит уже подготовленные данные сценария, нужные тестам и фикстурам.

Предлагаемые поля:

- `scenario`
- `phone`
- `user_id`
- `role`
- `expected_home_state`
- `country_code`
- `is_new_user`
- `selection_source`
- `description`

Примечания:

- `phone` — основное значение для логина, которое используют тесты
- `expected_home_state` позволяет фикстурам проверять ожидаемое состояние приложения без повторного кодирования логики сценария
- `selection_source` хранит источник выбора: CLI override, поиск в БД, найденный свободный номер и так далее
- `description` даёт понятное текстовое описание для логов и skip-сообщений

## Модель сценариев

На первом этапе стоит поддержать такие сценарии:

- `ONBOARDING_NEW_USER`
- `POTENTIAL_USER`
- `SUBSCRIBED_USER`
- `MEMBER_USER`
- `COACH_USER`
- `KYRGYZSTAN_ONBOARDING_NEW_USER`

Каждый сценарий определяет собственную стратегию выбора и ожидаемые выходные данные.

## Правила сценариев

### ONBOARDING_NEW_USER

Поведение:

- если передан `override_phone`, использовать его
- иначе подобрать свободный номер из настроенного диапазона для онбординга

Требования:

- номер не должен принадлежать существующему пользователю в БД
- результат трактуется как сценарий нового пользователя
- `expected_home_state` не обязателен до завершения онбординга

Предлагаемые метаданные:

- `country_code="+7"`
- `is_new_user=True`

### POTENTIAL_USER

Поведение:

- выбрать существующего пользователя, подходящего для mobile-flow в состоянии `NEW_USER`

Требования:

- `role == "potential"`
- `firstName` существует и не пустой
- у пользователя есть запись в `usermetadatas`
- номер телефона нормализуется в тот же формат, который используется в текущих login-flow

Предлагаемые метаданные:

- `expected_home_state=HomeState.NEW_USER`
- `is_new_user=False`

### SUBSCRIBED_USER

Поведение:

- выбрать существующего пользователя с активной подпиской

Требования:

- есть активная запись в `usersubscriptions`
- подписка не удалена
- это не детская подписка

Предлагаемые метаданные:

- `expected_home_state=HomeState.SUBSCRIBED`

### MEMBER_USER

Поведение:

- выбрать существующего пользователя с активным service product

Требования:

- есть активная запись в `userserviceproducts`
- service product не удалён
- это не child-продукт

Предлагаемые метаданные:

- `expected_home_state=HomeState.MEMBER`

### COACH_USER

Поведение:

- выбрать существующего coach-пользователя

Требования:

- есть запись в `coaches`
- связанный `user` существует и у него есть корректный номер телефона

Предлагаемые метаданные:

- `expected_home_state` можно пока не заполнять до появления полноценной поддержки coach shell

### KYRGYZSTAN_ONBOARDING_NEW_USER

Поведение:

- использовать отдельный номер для онбординга и country-specific login-flow

Требования:

- номер пока передаётся явно
- выбор страны является частью mobile-flow

Предлагаемые метаданные:

- `country_code="+996"`
- `is_new_user=True`

## Зоны ответственности

### MobileTestUserSelector

Отвечает за:

- приём запроса на сценарий
- применение правил CLI override
- вызов низкоуровневых repository helper-ов
- нормализацию результата в `TestUserContext`
- формирование единых диагностических сообщений, если подбор не удался

Не отвечает за:

- UI-действия
- выполнение логина или онбординга
- проверки экранов профиля

### Тесты и фикстуры

Тесты и фикстуры должны только:

- запросить контекст для нужного сценария
- использовать `context.phone`
- использовать `context.expected_home_state`, где это нужно

Они не должны:

- напрямую знать критерии отбора в БД
- самостоятельно выбирать fallback-кандидатов
- вшивать в себя диапазоны номеров для онбординга

## План интеграции

### tests/mobile/onboarding/test_client_onboarding.py

Убрать локальную логику выбора номера и заменить её вызовом селектора для сценария `ONBOARDING_NEW_USER`.

### tests/mobile/helpers/session_helpers.py

Заменить прямые вызовы `get_phone_for_potential_user(...)` и других подобных helper-ов на вызовы селектора для сценариев:

- `POTENTIAL_USER`
- `SUBSCRIBED_USER`
- `MEMBER_USER`
- `COACH_USER`

### tests/mobile/conftest.py

Сохранить CLI-фикстуры как источник сырых override-значений, но передавать их в селектор, а не заставлять тесты интерпретировать их самостоятельно.

### src/repositories/users_repository.py

Сохранить текущие низкоуровневые query helper-ы на первом этапе, но использовать их как внутренние детали реализации `MobileTestUserSelector`.

Такой подход позволит сделать миграцию постепенно, без большого одномоментного рефакторинга.

## Обработка ошибок

Неудачный выбор пользователя должен приводить к единообразным и завязанным на сценарий сообщениям.

Примеры:

- не найден свободный номер в диапазоне для онбординга
- не найден подходящий `potential`-пользователь с `usermetadatas`
- не найден кандидат с активной подпиской
- не найден coach-пользователь

Формирование таких сообщений должно быть сосредоточено в селекторе.

## Стратегия тестирования

Нужно добавить отдельные unit-тесты для нового модуля селектора.

Покрыть:

- приоритет CLI override
- подбор свободного номера для онбординга
- правила выбора `potential`-пользователя
- соответствие сценариев `subscribed/member/coach`
- нормализацию `TestUserContext`
- сценарии ошибок при отсутствии кандидатов

Существующие mobile-flow тесты продолжат проверять end-to-end поведение, а unit-тесты селектора покроют именно логику принятия решения.

## Заметки по миграции

Миграция должна быть поэтапной:

1. добавить новый модуль селектора
2. перевести на него onboarding flow
3. перевести session helper-ы и state-фикстуры
4. убрать или ослабить прямой выбор сценариев внутри тестов
5. при необходимости пометить старые публичные helper-точки как deprecated, когда они перестанут использоваться

## Финальные решения

Зафиксированные решения в рамках этого дизайна:

- централизуем не только выбор номера телефона, но и всю логику выбора mobile test user
- возвращаем объект контекста, а не просто номер телефона
- существующие repository helper-ы временно сохраняем как низкоуровневые примитивы
- основная публичная точка входа — отдельный модуль селектора

## Итог

Выбранный дизайн создаёт одно понятное место, которое отвечает на вопросы:

- под каким сценарием должен идти mobile-тест
- как подходящий пользователь находится в базе
- какой номер телефона нужно использовать
- какое состояние ожидается после логина

В результате границы ответственности становятся чище:

- репозитории делают низкоуровневый доступ к данным
- селектор владеет логикой выбора сценария
- фикстуры и тесты потребляют уже готовый контекст
