# Правила Allure-отчётов для backend-тестов

Этот документ задаёт единый стандарт оформления Allure-отчётов для backend-тестов. Его цель: чтобы новый тест сразу давал предсказуемый, читаемый и полезный отчёт без лишнего шума.

Документ прикладной: меньше теории, больше обязательных правил, шаблонов и примеров.

## 1. Что обязательно должно быть в каждом backend-тесте

Минимальный каркас нового backend-теста:

- `@allure.feature(...)` — верхнеуровневая бизнес-область.
- `@allure.story(...)` — проверяемый сценарий внутри области.
- `@allure.title(...)` — короткий бизнес-заголовок теста.
- `@allure.severity(...)` — критичность сценария.
- `@allure.tag(...)` — теги для фильтрации и группировки.
- `with allure.step(...)` — шаги по бизнес-смыслу, а не по реализации.
- `allure.dynamic.parameter(...)` — ключевые числа и параметры выборки.
- `allure.attach(...)` — summary, агрегаты, нарушения, диагностика.
- expected vs actual — в сообщении `assert` и, при наличии нарушений, в диагностическом вложении.
- последняя по времени ошибочная запись — должна быть выделена отдельно, чтобы было видно, сохраняется ли баг на текущем горизонте данных.
- общее число обработанных записей — должно быть видно всегда, даже если все записи успешные.
- данные могут и должны прикладываться в нескольких таблицах, если это помогает быстро увидеть проблему под разными углами.
- `assert ...` — с сообщением, которое помогает понять причину падения без чтения кода.

Что уже должно приходить автоматически через backend-инфраструктуру:

- окружение (`--backend-env`);
- период анализа (`period_days`).

Не дублируйте эти параметры вручную, если они уже централизованно прикладываются через `tests/backend/conftest.py`.
При этом период выборки должен быть явно виден в отчёте каждого backend-теста: как минимум в Allure parameters, а для summary-ориентированных сценариев ещё и в тексте summary или названии/тексте ключевого шага.

## 2. Как должен выглядеть хороший Allure-отчёт

### Заголовок теста

Хорошо:

- `При входе по абонементу accesscontrols.accessType=subscription`
- `Запрещённые типы транзакций не списывают бонусы`
- `Лимиты списания бонусов соблюдаются по типу абонемента`

Плохо:

- `Check access type`
- `Test payments`
- `Mongo validation`

Правило: заголовок должен описывать проверяемое бизнес-правило или ожидаемый инвариант.

### Шаги

Хорошо:

- `Вычислить нарушения accessType по абонементам`
- `Загрузить транзакции с bonusesSpent > 0 за 7 дней`
- `Приложить детализацию нарушений`

Плохо:

- `Call service`
- `Run query`
- `Process result`

Правило: шаг должен быть понятен QA, аналитику и разработчику без чтения исходников.

### Dynamic parameters

В `allure.dynamic.parameter(...)` выносим только то, что помогает быстро понять масштаб проверки:

- период выборки, если он не виден достаточно явно в уже прикреплённых параметрах;
- общее число обработанных записей;
- размер выборки;
- число найденных сущностей;
- число нарушений;
- число пропусков/исключений;
- ключевые агрегаты, влияющие на интерпретацию результата.

Не выносите туда шумовые значения, которые не помогают читать отчёт.
Правило для периода: читатель отчёта должен без открытия кода сразу видеть, за сколько дней или за какой интервал собраны данные.

## 3. Стандарт вложений

### Когда использовать `TEXT`

Используйте `TEXT` для:

- summary в человекочитаемом виде;
- агрегатов по клубам, планам, типам транзакций;
- списка нарушений;
- коротких диагностических блоков;
- компактных отчётов, которые удобно читать прямо в Allure.

Если тест работает по окну времени, в `Summary` должна быть явно указана выборка:

```text
Период: последние 7 дней
Обработано записей: 128
Нарушений: 3
```

Даже если нарушений нет, `Summary` всё равно должен отвечать на вопрос: сколько всего записей было обработано. Например:

```text
Период: последние 7 дней
Обработано записей: 5
Нарушений: 0
```

Примеры имён:

- `Summary`
- `By Club`
- `By Subscription Plan`
- `Нарушения по клубам`
- `Транзакции без PAY-записи`

### Когда использовать `HTML`

Используйте `HTML` для:

- таблиц, которые выигрывают от визуального форматирования;
- агрегатов с несколькими колонками;
- детализации нарушений, если plain text уже неудобен.
- разных представлений одной и той же выборки, если каждое представление отвечает на отдельный вопрос анализа.

Во всех аналитических таблицах по возможности показывайте не только счётчики, но и идентифицирующие и человекочитаемые поля:

- имена сущностей;
- названия клубов, продуктов, планов, сервисов;
- `id` / `_id` / `tx_id` / `entry_id` и другие ключевые идентификаторы;
- типы (`clubType`, `productType`, `accessType`, `kind` и т.д.);
- даты и время;
- количество записей, число нарушений и другие важные счётчики.

Правило: таблица должна помогать понять проблему без обращения к коду, MongoDB или дополнительным логам.

Примеры имён:

- `Summary (HTML)`
- `By Club (HTML)`
- `Нарушения по клубам (HTML)`

Не ограничивайтесь одной “универсальной” таблицей, если проблема читается быстрее через несколько срезов. Нормальный набор представлений для одного и того же отчёта может включать:

- таблицу по клубам;
- таблицу по типам продукта или транзакции;
- таблицу по expected vs actual;
- таблицу только по последним ошибкам;
- таблицу по корзинам/диапазонам значений;
- таблицу по критичности или типу нарушения.

Правило: каждая дополнительная таблица должна отвечать на конкретный вопрос. Например:

- `By Club (HTML)` — в каких клубах проблема концентрируется;
- `By Product Type (HTML)` — к каким типам продукта проблема привязана;
- `Expected vs Actual (HTML)` — какие именно расхождения встречаются чаще всего;
- `Latest Violations (HTML)` — есть ли свежие ошибки прямо сейчас.

Если один HTML-репорт не даёт быстрого ответа на эти вопросы, лучше приложить 2-4 компактные таблицы, чем одну перегруженную.
При этом в каждой таблице старайтесь держать баланс между наглядностью и идентификацией: например, `clubName + clubId`, `planName + planId`, `expected + actual + tx_id`, `latest time + tx_id + userId`.

### Отчёты для уведомлений

В каждом Allure-отчёте для теста уведомлений должен быть отдельный пункт с описанием параметров, по которым тест считается пройденным. Достаточно обычного текстового вложения `Pass criteria` с типом `TEXT`.

В `Pass criteria` кратко фиксируйте проверяемые параметры и ожидаемые значения: кому должно прийти уведомление, какой тип/шаблон ожидается, какие языки/локали должны быть покрыты, какие поля уведомления проверяются, какие значения считаются корректными и какие нарушения приводят к падению теста. Этот пункт должен отвечать на вопрос: по каким параметрам результат теста признаётся успешным.

Для notification-сценариев с несколькими языками допустимо и рекомендуется добавлять отдельное HTML-вложение `Language summary`, как в `tests/backend/notifications/test_guest_visit_discount_push.py`.

Правило для `Language summary`: по каждому языку нужно показывать не только агрегат, но и до 10 репрезентативных примеров. Такой отчёт должен отвечать сразу на два вопроса: сколько документов найдено по каждому языку и как выглядят реальные примеры внутри каждого языкового варианта.

Предпочтительный состав `Language summary`:

- язык;
- число документов или событий по этому языку;
- до 10 примеров для каждого языка;
- поля примеров, которые помогают быстро понять содержимое уведомления: например `title`, `text`, `created_at`, `description`, `userId` или другие релевантные идентификаторы.

Если по языку найдено больше 10 записей, в `Language summary` показывайте только первые 10 или самые показательные 10, а полный массив при необходимости выносите в отдельную диагностику. Не перегружайте `Language summary` всеми записями сразу.

### Когда использовать `JSON`

Используйте `JSON` только когда он действительно нужен для отладки:

- полный сырой результат проверки;
- выборка записей, по которым сложно восстановить контекст из текста;
- снимок данных, который нужен для повторного анализа.

Не прикладывайте JSON по умолчанию в каждый тест, если текстового и HTML-отчёта достаточно.

### Какой набор вложений считается хорошим

Для типового backend-check сценария ориентируемся на такой порядок:

1. Summary.
2. Агрегаты по важным срезам.
3. Детализация нарушений, если нарушения есть.
4. Expected vs actual для спорных или массовых нарушений.
5. Последняя по времени ошибочная запись.
6. Дополнительные таблицы по альтернативным срезам, если они ускоряют анализ.
7. Диагностический блок только если он помогает объяснить спорные случаи.
8. JSON только если без него расследование будет затруднено.

Если тест сравнивает ожидаемое и фактическое значение, это сравнение должно быть видно в отчёте явно, а не только подразумеваться из текста ошибки.
Если тест анализирует данные за период, этот период должен быть виден в верхней части отчёта без необходимости открывать код или fixtures.
Если нарушения найдены, читатель отчёта должен сразу видеть самую свежую ошибочную запись с её временем, чтобы понять, баг исторический или всё ещё актуальный.
Даже при полностью успешной проверке отчёт должен показывать объём обработанных данных, чтобы было видно, что тест действительно что-то проверил.
Если для понимания проблемы нужны разные ракурсы, в отчёте должно быть несколько таблиц с разными группировками, а не одна таблица “на всё”.

Хорошо:

```text
- entry_id=abc123 | expected=subscription | actual=visit | club=Invictus Astana
```

Плохо:

```text
- entry_id=abc123 | accessType неверный
```

Хорошо:

```text
| clubName | clubId | productType | records | violations |
| Invictus Astana | 67f... | visits | 15 | 3 |
```

Плохо:

```text
| club | count |
| 67f... | 3 |
```

Хорошо:

```text
Последняя ошибочная запись
- 2025-04-15 09:41:22 | entry_id=xyz789 | expected=subscription | actual=visit | club=Invictus Astana
```

Плохо:

```text
Нарушения есть, см. список выше
```

## 4. Формат текстовых вложений

Главное правило: текстовые вложения должны быть компактными.

Предпочтительный формат:

- одна строка на одну сущность;
- поля разделяются через `|`;
- записи группируются под коротким заголовком сущности или клуба.

Хорошо:

```text
Invictus Astana (3)
- 2025-01-01 12:00:00 | trans_id=abc123 | 5 000 тг | неверный accessType=visit
- 2025-01-01 13:10:00 | trans_id=def456 | 7 000 тг | отсутствует PAY-запись
```

Плохо:

```text
Клуб: Invictus Astana
Транзакция: abc123
Дата: 2025-01-01 12:00:00
Сумма: 5 000 тг
Проблема: неверный accessType=visit

Клуб: Invictus Astana
Транзакция: def456
Дата: 2025-01-01 13:10:00
Сумма: 7 000 тг
Проблема: отсутствует PAY-запись
```

Правило: если сущностей много, многострочные карточки резко ухудшают читаемость.

## 5. Что запрещено

- Не использовать vague title без бизнес-смысла.
- Не писать технические `step`, понятные только автору теста.
- Не падать без контекста: при нарушениях должен быть отчёт или диагностическое вложение.
- Не скрывать expected vs actual внутри кода или только в голове автора теста.
- Не оставлять читателя без ответа на вопрос, есть ли свежая ошибка на текущий момент.
- Не оставлять отчёт без ответа на вопрос, сколько записей было обработано всего.
- Не сводить сложную проблему к одной таблице, если для анализа нужны разные срезы.
- Не строить аналитические таблицы только из count-полей без имён, id, типов и других поясняющих колонок.
- Не прикладывать шумовые вложения, которые не помогают расследованию.
- Не смешивать бессистемные имена вложений вроде `report`, `data`, `test1`, `debug_info`.
- Не дублировать один и тот же смысл в трёх форматах без причины.
- Не прикладывать огромный JSON, если ту же проблему уже объясняет компактный `TEXT`.

## 6. Эталонная структура backend-теста

Короткий пример по мотивам `tests/backend/payments/test_subscription_access_type.py` и похожих backend-check сценариев:

```python
import pytest
import allure

from src.services.backend_checks.payments_checks_service import run_subscription_access_type_check
from src.services.reporting.payments_text_reports import build_subscription_access_type_reports


@pytest.mark.backend
@allure.feature("Access Control")
@allure.story("Subscription entry validation")
@allure.title("При входе по абонементу accesscontrols.accessType=subscription")
@allure.severity(allure.severity_level.CRITICAL)
@allure.tag("backend", "accesscontrols", "subscription", "access")
def test_subscription_entry_access_type(db, period_days):
    with allure.step("Вычислить нарушения accessType по абонементам"):
        result = run_subscription_access_type_check(db=db, period_days=period_days, sample_size=300)

    if result.entries_count == 0:
        pytest.skip(f"Нет входов по абонементам за последние {period_days} дней")

    allure.dynamic.parameter("Период выборки", f"последние {period_days} дней")
    allure.dynamic.parameter("Обработано записей", result.entries_count)
    allure.dynamic.parameter("Входов в выборке", result.entries_count)
    allure.dynamic.parameter("Входов в окне абонемента", result.entries_in_window_count)
    allure.dynamic.parameter("Нарушений", len(result.violations))

    reports = build_subscription_access_type_reports(result)

    with allure.step("Приложить summary и агрегаты"):
        allure.attach(
            f"Период: последние {period_days} дней\n"
            f"Обработано записей: {result.entries_count}\n"
            + reports["summary_text"],
            name="Summary",
            attachment_type=allure.attachment_type.TEXT,
        )
        allure.attach(reports["summary_html"], name="Summary (HTML)", attachment_type=allure.attachment_type.HTML)
        allure.attach(reports["by_club_text"], name="By Club", attachment_type=allure.attachment_type.TEXT)
        allure.attach(reports["by_club_html"], name="By Club (HTML)", attachment_type=allure.attachment_type.HTML)
        allure.attach(
            reports["by_access_type_html"],
            name="By Actual Access Type (HTML)",
            attachment_type=allure.attachment_type.HTML,
        )

    if reports["violations_text"]:
        with allure.step("Приложить детализацию нарушений"):
            allure.attach(
                reports["violations_text"],
                name="Нарушения по клубам",
                attachment_type=allure.attachment_type.TEXT,
            )
            allure.attach(
                reports["violations_html"],
                name="Нарушения по клубам (HTML)",
                attachment_type=allure.attachment_type.HTML,
            )
            allure.attach(
                reports["expected_vs_actual_text"],
                name="Expected vs Actual",
                attachment_type=allure.attachment_type.TEXT,
            )
            allure.attach(
                reports["latest_violation_text"],
                name="Последняя ошибочная запись",
                attachment_type=allure.attachment_type.TEXT,
            )
            allure.attach(
                reports["latest_violations_html"],
                name="Latest Violations (HTML)",
                attachment_type=allure.attachment_type.HTML,
            )

    assert len(result.violations) == 0, (
        f"Найдено {len(result.violations)} входов по абонементу с accessType != 'subscription'. "
        f"Последняя ошибочная запись: entry_id={result.latest_violation.entry_id}, "
        f"timestamp={result.latest_violation.created_at}, "
        f"expected='subscription', actual='{result.latest_violation.access_type or '—'}'"
    )
```

Что этот пример показывает:

- декораторы задают бизнес-контекст теста;
- шаги отражают смысл проверки;
- dynamic parameters показывают период выборки, общий объём обработанных данных, масштаб и итог;
- вложения разделены на summary, агрегаты, нарушения, expected vs actual, последнюю ошибочную запись и дополнительные аналитические таблицы;
- `assert` объясняет, что именно сломалось, и явно показывает самую свежую ошибку.

## 7. Чеклист перед добавлением нового backend-теста

Перед коммитом проверьте:

- заголовок теста описывает бизнес-правило;
- шаги понятны без чтения кода;
- период выборки явно виден в отчёте;
- общее число обработанных записей видно даже при успешном результате;
- в отчёте есть summary или эквивалентный краткий итог;
- при наличии нарушений прикладывается читаемая детализация;
- expected vs actual явно видны в диагностике для каждого важного нарушения;
- при наличии нарушений отдельно выделена последняя по времени ошибочная запись;
- при необходимости данные показаны в нескольких таблицах с разными срезами для быстрого анализа;
- в аналитических таблицах есть имена, записи, id, типы и другие поясняющие колонки, а не только сухие счётчики;
- текстовые вложения компактны и используют `|`-разделители;
- HTML используется только там, где таблица реально улучшает чтение;
- JSON прикладывается только при диагностической ценности;
- для тестов уведомлений есть текстовое вложение `Pass criteria` с параметрами успешного прохождения;
- сообщение в `assert` помогает понять причину падения сразу из Allure.
